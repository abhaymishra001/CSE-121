🔄 Introduction to DevOps Section 1
What is DevOps?
📌 Definition
DevOps = Development (Dev) + Operations (Ops). It is a software development approach that emphasizes collaboration and communication between development and operations teams. DevOps is a set of practices and tools designed to shorten the life cycle of a software development process while delivering high-quality software.
  • Allows a single team to handle the entire application lifecycle — from development to testing, deployment, and operations
  • Reduces the disconnection between software developers, quality assurance engineers, and system administrators
  • Automates workflows for faster delivery
  • Enables continuous integration and continuous deployment (CI/CD)
  • Boosts software quality, speed, and security
  • Encourages iterative feedback and constant improvement
DevOps vs Traditional Software Development
Traditional Software Models
  • Waterfall: Sequential, rigid, phase-by-phase approach — each phase must complete before the next begins
  • Iterative: Incremental, repetitive cycles with feedback loops
  • Agile: Flexible, customer-centric approach with short sprints
  • V-Model: Verification and validation at each stage
AspectTraditionalDevOps
Team StructureSeparate Dev and Ops teamsUnified, cross-functional team
DeploymentManual, infrequent releasesAutomated, continuous deployment
FeedbackSlow feedback loopsContinuous, real-time feedback
TestingEnd-of-cycle testingContinuous testing throughout
SpeedSlower deliveryFaster, iterative delivery
RiskHigher risk at deploymentLower risk, incremental changes
Real-Life Applications of DevOps
  • Online Financial Trading: Automated testing, building & development — deployment in 45 seconds instead of long nights
  • Network Cycling: Deployment and testing became 10x faster; security patches daily instead of quarterly
  • Car Manufacturing: DevOps helped catch errors while scaling production
  • Airline Industry: United Airlines saved $500,000 by switching to continuous testing; code coverage increased by 85%
  • GM Financial: Regression testing time reduced by 93%; loan funding period reduced 5x
  • Key Bank: Integration of security and compliance reduced from 3 months to 1 week

DevOps = Dev + Ops = Collaboration + Automation + CI/CD + Continuous Improvement. It's NOT just a tool — it's a culture and practice.
🔧 DevOps Tools Section 2
📦 Git — Version Control
Type: Distributed version control system
Purpose: Track code changes, enable collaboration
Key Features: Branching, merging, rollback, commit history
Platforms: GitHub, GitLab, Bitbucket
Stat: 90%+ developers use Git (Stack Overflow Survey)
Benefits: Parallel development, code review, efficient history management
🐳 Docker — Containerization
Type: Containerization platform
Purpose: Develop, ship, run apps in containers
Architecture: Dockerfile → Image → Container → Docker Hub
Benefits: Isolation, portability, scalability, consistency
Key: Apps work the same in dev, test, and production environments
🧪 Selenium — Automated Testing
Type: Portable framework for web application testing
Features: Cross-browser compatibility, multi-language support, record/playback
Benefits: Automates web testing, ensures reliability, reduces manual regression effort
🏗️ Maven — Build Automation
Type: Build automation tool for Java projects
Features: Dependency management, project build lifecycle, plugin support
Uses POM files to manage project structure
Automates: Compiling, testing, packaging software
🤖 Puppet & Ansible — Config Management
Puppet: Agent-based, declarative language for configuration
Ansible: Agentless, YAML-based — simpler setup
Use Cases: Infrastructure as Code (IaC), automated deployments, compliance management
Ansible Benefits: Automates IT tasks, manages complex workflows, no agent installation needed
☸️ Kubernetes — Container Orchestration
Type: Open-source container orchestration system
Purpose: Automate deployment, scaling, management of containerized apps
Components: Pods, Deployments, Services, Nodes
Enables: High availability, scalability, resilience, self-healing

Nagios is a comprehensive IT infrastructure monitoring tool. It monitors servers, services, and networks. It receives alerts and generates reports to identify and resolve issues proactively. Functionality can be extended with Nagios plugins. Monitors: Server health, Service availability, Network performance.

♻️ DevOps Lifecycle Section 3

The DevOps lifecycle is a continuous cycle of 8 phases that ensure rapid, reliable software delivery:

1. Plan
2. Develop
3. Build
4. Test
5. Release
6. Deploy
7. Operate
8. Monitor
Phase Details
Planning: Define project goals, scope, requirements, milestones. Task allocation using Jira, Trello, Asana. Align dev and ops teams.
Coding: Developers write clean, maintainable code. Version control using Git, GitLab, Bitbucket. Ensures collaboration and conflict management.
Build: Source code compiled into executable/deployable artifacts. Tools: Maven, Gradle, Jenkins. Automation reduces errors.
Testing: Automated testing using Selenium, JUnit, TestNG. Detects bugs early. Ensures code quality and reliability.
Release: Package and prepare application for production. Version control and automation using Jenkins, Ansible, Docker.
Deployment: Application deployed to production. Automation ensures minimal downtime. Containerization with Docker & orchestration via Kubernetes.
Operate: Manage servers, networks, uptime, infrastructure. Tools: Nagios, Splunk, New Relic. Ensures system stability.
Monitoring: Continuous monitoring of logs, metrics, user behavior. Tools: Prometheus, Grafana, ELK Stack. Provides feedback for improvement.
Role of Automation in DevOps
  • CI (Continuous Integration): Automated build & testing — Jenkins, GitLab
  • CD (Continuous Deployment): Automated release & deployment — Docker, Kubernetes
  • IaC (Infrastructure as Code): Automated provisioning — Terraform, CloudFormation
  • Automated Testing: Selenium, JUnit for early bug detection
  • Monitoring & Alerts: Prometheus, Grafana for real-time insights
Benefits of Automation
  • Faster software delivery and shorter release cycles
  • Reduced human errors and improved reliability
  • Continuous feedback and performance monitoring
  • Improved collaboration between Dev & Ops teams
  • Scalable, consistent, and repeatable infrastructure setup
🚀 CI/CD Pipeline Section 4
📌 Definition
CI/CD stands for Continuous Integration and Continuous Delivery/Deployment. It aims to streamline and accelerate the software development lifecycle through automation.
Continuous Integration (CI)

CI is the practice of automatically integrating code changes into a shared repository frequently — daily or multiple times per day.

Key Activities
  • Developers commit code to a shared repository (e.g., Git)
  • Automated build process runs automatically
  • Automated unit and integration tests execute
  • Immediate feedback is provided to developers
Benefits of CI
  • Early bug detection
  • Reduced integration problems
  • Improved collaboration
  • Higher code quality
Continuous Delivery vs Continuous Deployment
AspectContinuous DeliveryContinuous Deployment
DefinitionCode is always in a deployable state; release requires manual approvalEvery validated change is automatically released to production
Human InterventionManual approval before productionNo manual approval — fully automated
BenefitFaster and safer releasesRapid innovation, faster time-to-market
Risk LevelLower (human checkpoint)Requires robust automated testing

🧪 Fundamentals of Software Testing Section 5
📌 Definition
Software Testing is the process of verifying and validating that a software application is free of bugs, meets technical requirements, and satisfies user requirements efficiently and effectively. The cost of software bugs reached $1.7 trillion in 2022.
Objectives of Testing
  1. Ensures software meets user requirements
  2. Identifies bugs early in the development process
  3. Improves product quality and overall reliability
  4. Reduces risk of software failure
  5. Validates functionality, performance, and security
  6. Supports continuous integration and delivery pipelines
Types of Software Testing
⬛
Black Box Testing
Test without knowing internal code. Focus on inputs and outputs. Examples: Functional & UI testing. Tester doesn't need programming knowledge.
⬜
White Box Testing
Test WITH knowledge of code. Focus on code structure and logic. Examples: Unit Testing & Code Coverage. Requires programming skills.
🔲
Gray Box Testing
Hybrid approach — combines Black Box and White Box techniques. Tester has partial knowledge of internal workings.
📝 Other Testing Types
Smoke Testing: Quick check if basic features work. Sanity Testing: Verify specific bug fixes. Regression Testing: Ensure new changes don't break existing features. Performance Testing: Test speed & scalability. Security Testing: Find security vulnerabilities.
Levels of Software Testing
Unit Testing
Integration Testing
System Testing
Acceptance Testing
  • Unit Testing: Test individual components/functions in isolation
  • Integration Testing: Test interaction between integrated units/modules
  • System Testing: Test the entire system as a whole
  • Acceptance Testing: End-users test for acceptability — the final checkpoint before release
Applications of Software Testing
  • Software Development: Ensure quality for web, mobile, desktop apps
  • Embedded Systems: Test software in cars, appliances, IoT devices
  • Gaming Industry: Test game functionality and performance
  • Healthcare: Test medical devices for accuracy and safety
  • Finance: Guarantee security for financial software and transactions

Introduction to Test Case Design
📌 What is a Test Case?
A test case is a documented set of conditions, inputs, actions, and expected results that verifies whether a software application works as expected.
Components of a Test Case
  • Test Case ID: Unique identifier (e.g., TC_LOGIN_001)
  • Description: Brief summary of what's being tested
  • Preconditions: Required system state before testing
  • Test Steps: Detailed, sequential actions to perform
  • Test Data: Specific inputs (e.g., username, password)
  • Expected Result: What should happen if software works correctly
  • Actual Result: Observed outcome (filled during execution)
  • Status: Pass or Fail
Example: Login Test Case
FieldDetails
Test Case IDTC_LOGIN_001
DescriptionVerify successful login with valid credentials
PreconditionsUser has registered account, is on login page
Test Steps1. Enter valid email 2. Enter valid password 3. Click "Login"
Expected ResultUser redirected to dashboard with "Welcome" message
StatusPass/Fail
Test Case Design Techniques
  • Boundary Value Analysis (BVA): Test edges of input ranges (if field accepts 1–100, test 0, 1, 100, 101)
  • Equivalence Partitioning: Divide inputs into groups with similar behavior, test one from each
  • Decision Table Testing: Map input combinations to expected outputs for complex logic
  • Error Guessing: Predict bugs based on experience and common pitfalls
🐛 Defect Life Cycle Section 6
📌 Definition
The Defect Life Cycle (also called Bug Life Cycle) describes the journey of a defect from identification to closure. It ensures proper tracking of bugs, clear communication between testers and developers, and improved software quality.
Stages of the Defect Life Cycle
1. New
2. Assigned
3. Open
4. Fixed
5. Retest
6. Verified
7. Closed
  1. New: Tester identifies a defect and logs it in a tracking tool (e.g., Jira)
  2. Assigned: The defect is assigned to a developer for fixing
  3. Open: Developer starts analyzing and working on the defect
  4. Fixed: Developer fixes the defect and updates the status
  5. Retest: Tester retests the defect to verify the fix
  6. Verified: Tester confirms the defect is resolved correctly
  7. Closed: If working correctly, the defect is marked as closed
Other Possible States
StatusMeaning
ReopenedDefect still exists after attempted fix
RejectedNot a valid defect — works as per requirement
DuplicateSame defect already reported by someone else
DeferredFix postponed to a future release (low priority)
Not a BugBehavior is as per requirement — not a defect
Example — Login System Defect
🔧 Scenario
Problem: Username field accepts 5–10 characters, but system allows 3-character usernames.
Flow: Tester logs defect → New → Assigned to developer → Assigned → Developer fixes validation → Fixed → Tester retests → Verified → Defect marked → Closed
Why is the Defect Life Cycle Important?
  • Ensures accountability — every defect has an owner
  • Improves communication between testers and developers
  • Maintains defect history for future reference
  • Supports quality assurance process
  • Helps in project tracking and timeline management

✏️ Quick Revision Revision
Q: What is DevOps?
Dev + Ops = Collaboration between development and operations teams with automation and CI/CD for faster, reliable software delivery.
Q: Name the 8 DevOps lifecycle phases.
Plan → Develop → Build → Test → Release → Deploy → Operate → Monitor (continuous cycle).
Q: What is CI/CD?
CI = Automatic code integration + testing. CD = Automatic deployment (Delivery needs approval, Deployment is fully automatic).
Q: What does Docker do?
Containerization — packages apps with dependencies so they run consistently in any environment.
Q: What does Kubernetes do?
Container orchestration — automates deployment, scaling, and management of containerized apps.
Q: Difference between Black & White Box?
Black Box = test without code knowledge. White Box = test with code knowledge. Gray Box = partial knowledge.
Q: Levels of testing?
Unit → Integration → System → Acceptance Testing (smallest to largest scope).
Q: Defect life cycle stages?
New → Assigned → Open → Fixed → Retest → Verified → Closed. Others: Reopened, Rejected, Deferred.
Q: Manual vs Automation testing?
Manual = human, slower, good for UI. Automation = scripts, faster, good for regression. Both are needed.
Q: What is BVA?
Boundary Value Analysis — test edges of input ranges (if 1-100, test 0, 1, 100, 101).
